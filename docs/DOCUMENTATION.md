# ドキュメント作成ルール

## 構成

- ユーザー向けの内容（メリット・使い方）と開発者向けの内容（仕組み・実装詳細）を分離する
- 同じ情報を複数のファイルに書かない。参照で済む場合は参照にとどめる

## 文体

- 誇張表現を避け、技術的に正確な表現を使う
  - 例: 「リソースを消費しません」→「定数時間で判定できます」
- 曖昧な表現を避け、具体的に何が起こるかを明記する
  - 例: 「勝手に最適化を行います」→ 何をどう最適化するのかを書く

## 言語

- セクション名やコメントなど、日本語で書ける部分は日本語で書く
  - ただし、正式名称や一般的な用語が英語表記の場合はそのまま使う（例: GitHub Actions、ESM、Tree Shaking）

## 表記

- ツール・サービス・技術用語は正式名称を使う
  - 例: Github → GitHub、typescript → TypeScript、npm → npm（小文字が正式）
- 半角英数字と日本語の間に半角スペースを入れる
  - 例: 「TypeScript ライブラリ」「ESM のみ」
- 括弧や記号は半角を使う
  - 例: `(``)` `—` ではなく全角の `（``）`や `ー` を使わない
- コード、コマンド、ファイルパスはバッククォートで囲む
  - 例: `dist/`、`npm run build`、`YYYY-MM-DD`

## README.md

- 「特徴」セクションはユーザーが得られるメリットを簡潔に列挙する。実装の詳細は書かない
- 「仕組み」セクションで、特徴がなぜ実現できるのかを技術的に説明する
